# 通用设计原则（Universal Principles）

> 定位：判断库，回答"怎样才算好"。
> 读者：AI Skill 执行时的约束引擎，不是给人读的哲学金句。
> 一条原则如果不能转化为 AI 可判定的条件，就不该入库。

---

## 原则卡片结构说明

每条原则一张卡片，固定 7 个字段：

| 字段 | 作用 | 为什么不能省 |
|---|---|---|
| 编号 | 被引用的 ID | 业务原则需要写"继承自 U-xxx" |
| 一句话 | 人类快速理解 | 设计师浏览时快速定位 |
| 判定条件 | AI 执行的依据 | 这是原则层最核心的字段，没有它原则就是空话 |
| 适用场景 | 限定作用范围 | 避免原则被滥用到不相关场景 |
| 与谁冲突 | 预声明冲突 | 让 AI 知道何时需要调用取舍准则 |
| 冲突时默认立场 | 给出默认解 | 没有业务原则覆写时的兜底判断 |
| 正例/反例 | 具象化理解 | 帮 AI 理解边界情况 |

## U-001 费茨定律

**一句话：** 目标越大、距离越近，操作越快。

**判定条件：**
- 可点击/可触摸元素的尺寸 ≥ 44px × 44px
- 高频操作入口距拇指热区 ≤ 2 次滑动
- 主要 CTA 按钮宽度 ≥ 屏幕宽度的 40%
- 间距 ≤ 8px 的相邻可点击元素视为一个操作区域
- 核心操作按钮必须位于屏幕底部 1/3 区域（移动端拇指可达区）
- 输入框获焦弹出虚拟键盘时，页面布局必须自适应，确保输入框与提交按钮不被键盘遮挡
- 移动端页面内容区不得依赖水平滚动展示核心信息

**适用场景：** 所有涉及点击、触摸、拖拽操作的界面元素。

**与谁冲突：**
- U-003 席克定律：大按钮可能增加视觉权重，使选项感知数量增加
- U-004 峰终定律：为营造峰值体验，可能需要缩小非核心操作突出核心
- U-007 安全确认：安全场景可能需要故意增加操作难度
- U-014 关键信息前置：重要信息在前 vs. 核心操作在底部——两者不矛盾（信息在顶部供扫描，操作在底部供触达），但需确保首屏同时可见关键信息与操作入口（○ 偶尔冲突）

**冲突时默认立场：** 优先保证可触达性，除非业务原则显式覆写。

**正例：** 充话费按钮 48px 高，位于首屏拇指热区，一键直达。
**反例：** 付款按钮 28px 高，需要滑动页面才能触及。

## U-002 格式塔原则

**一句话：** 元素之间的关系由接近性、相似性、闭合性和连续性决定。

**判定条件：**

| 子原则 | 判定条件 |
|---|---|
| 接近性 | 同组元素间距 ≤ 8px，不同组间距 ≥ 24px |
| 相似性 | 同功能的元素使用相同的视觉样式（颜色、形状、大小） |
| 闭合性 | 卡片/区域必须有明确的视觉边界（边框、背景色、阴影至少一种） |
| 连续性 | 对齐方式在同一行/列内保持一致；同类信息使用相同的排列方向 |

**适用场景：** 所有涉及元素分组、布局、对齐的界面。

**与谁冲突：**
- U-005 米勒定律：分组过多可能增加信息组块数量（○ 偶尔冲突）

**冲突时默认立场：** 优先遵循格式塔的分组逻辑，当分组数量超过米勒定律限制时，合并或折叠次要分组。

**正例：** 充值金额选项以 3×2 网格排列，同组间距 8px，不同组间距 24px，选中的金额用主题色高亮。
**反例：** 充值金额和功能入口混在一起，间距相同，无法区分哪些是金额选项。

## U-003 席克定律

**一句话：** 选项越少，决策越快；每多一个选项，决策时间对数增长。

**判定条件：**
- 单屏主操作按钮 ≤ 5 个
- 单屏决策选项 ≤ 7 个（超过必须分组/折叠/分步）
- 每个决策点只呈现与当前步骤相关的选项
- 下一步操作必须能在 3 秒内识别

**适用场景：** 所有涉及用户选择和决策的界面。

**与谁冲突：**
- U-001 费茨定律：减少选项可能使保留的选项更大，两者通常协同而非冲突
- U-005 米勒定律：减少选项是米勒定律的手段之一，两者方向一致但角度不同——席克管"决策数量"，米勒管"信息组块"
- U-009 识别而非回忆：显示所有选项 vs 隐藏次要选项（○ 偶尔冲突）

**冲突时默认立场：** 优先减少决策选项，但不得隐藏用户必需的信息。当选项必须全部可见时，通过视觉分组降低决策复杂度。

**正例：** 充值页面只展示 3 个常用金额 + 1 个自定义入口，共 4 个选项。
**反例：** 充值页面展示 10 个金额选项，无分组，用户需要逐个比较。

## U-004 峰终定律

**一句话：** 用户对体验的记忆由最强烈的瞬间和结束时的感受决定。

**判定条件：**
- 多步流程（≥ 3 步）必须有至少 1 个明确的峰值时刻（超出预期的正面体验）
- 流程的最后一步必须以正向状态结束（成功提示/庆祝动画/结果确认）
- 峰值时刻必须与核心价值相关，不能是无意义的装饰
- 流程中不得出现"平"的结束——没有确认的提交、没有反馈的等待

**适用场景：** 所有多步骤流程（充值、申请、汇款、开通等）。

**与谁冲突：**
- U-001 费茨定律：为营造峰值可能需要调整非核心操作的可触达性
- U-006 反馈即时性：峰值体验可能需要延迟反馈以制造惊喜感（○ 偶尔冲突）
- U-008 一致性：为峰值体验可以打破流程间的一致性

**冲突时默认立场：** 峰终体验优先于一致性，但不得牺牲可触达性和即时反馈。

**正例：** 充值成功页用动画确认到账金额，并在底部推荐下一步常用操作（正向终值 + 引导延续）。
**反例：** 充值成功后直接返回首页，无任何确认，用户不确定是否到账。

## U-005 米勒定律（7±2）

**一句话：** 工作记忆同时处理的信息不超过 7±2 个组块。

**判定条件：**
- 单屏可见信息组块 ≤ 7 个
- 超过 5 个组块时必须使用视觉分组（标题/分割线/卡片）
- 超过 7 个组块时必须使用折叠、分页或分步
- 单个列表项包含的信息字段 ≤ 5 个

**适用场景：** 所有信息展示页面。

**与谁冲突：**
- U-002 格式塔原则：格式塔的分组可能增加感知组块数（○ 偶尔冲突）
- U-003 席克定律：减少选项和减少组块方向一致，但席克管"决策"，米勒管"认知"（● 强冲突场景：业务要求展示很多信息但用户不需要全部决策）
- U-010 自然语言表达：用户习惯的信息呈现方式可能组块较多（○ 偶尔冲突）

**冲突时默认立场：** 信息组块数量不得违反 7±2 上限，通过分组、折叠、渐进式展示来容纳更多信息。

**正例：** 信用卡账单页将信息分为 3 个卡片：本月应还、最近交易（折叠）、还款入口，共 3 个组块。
**反例：** 信用卡账单页平铺 12 个信息项，无分组无折叠，用户需要逐行扫描。

## U-006 反馈即时性

**一句话：** 用户操作后 200ms 内必须有可感知的反馈，交互完整闭环 ≤ 400ms。

**判定条件：**
- 按钮/开关点击后 ≤ 200ms 出现视觉状态变化（按下态/加载态）
- 从操作到完整结果呈现 ≤ 400ms（Doherty Threshold：超过此阈值用户感知为"等待"，生产力骤降）
- 表单提交后 ≤ 200ms 出现加载指示器
- 页面切换时使用骨架屏或加载占位符，不得出现空白屏
- 加载时间 > 1s 时必须显示进度指示
- 加载时间 > 3s 时必须提供可操作的其他入口
- 交互响应超过 400ms 时必须提供中间状态反馈（加载动画、进度条、骨架屏），不得让用户面对无变化界面

**适用场景：** 所有用户交互场景。

**与谁冲突：**
- U-004 峰终定律：峰值体验可能需要刻意延迟以制造惊喜（○ 偶尔冲突）
- U-007 安全确认：需要二次确认的操作无法即时完成

**冲突时默认立场：** 反馈即时性不可妥协，即使在需要二次确认的场景中也必须先给出"已收到操作"的即时反馈，再进入确认流程。

**正例：** 点击"立即充值"后，按钮立即变为加载态（200ms 内），同时弹出确认弹窗，确认后 350ms 内显示结果页。
**反例：** 点击"立即充值"后，页面无任何变化，2 秒后弹窗才出现。

## U-007 安全确认与容错

**一句话：** 涉及资金或不可逆操作时，必须增加二次确认；出错后必须提供诊断和恢复路径。

**判定条件：**
- 涉及金额支付/转账的操作 → 必须有确认弹窗，显示金额和收款方
- 不可逆操作（删除、取消、退订）→ 必须有确认弹窗，说明不可逆后果
- 确认弹窗的操作按钮必须使用明确的行为动词（"确认支付 ¥50"而非"确定"）
- 确认弹窗的取消按钮必须可见且易于点击
- 不得使用暗黑模式（默认勾选、隐藏取消按钮、倒计时自动确认）
- 可撤销操作（编辑、加入购物车等）→ 必须提供明确的撤销入口（"撤销"按钮或 Ctrl+Z），撤销窗口 ≥ 3 秒
- 不可撤销操作出错后 → 必须提供错误诊断信息（发生了什么 + 为什么 + 下一步怎么办），不得只显示错误码
- 错误提示必须包含可操作的恢复建议（"请检查网络后重试"/"联系客服 400-xxx"），不得止步于"操作失败"

**适用场景：** 所有涉及资金、个人信息、不可逆操作的界面，以及所有错误状态页面。

**与谁冲突：**
- U-001 费茨定律：确认步骤增加了操作路径长度
- U-006 反馈即时性：确认流程阻碍了即时完成
- U-003 席克定律：撤销/重做入口增加界面选项（○ 偶尔冲突）
- U-015 用户控制与自由：安全场景需要限制用户自由（如支付已提交不可撤销）——此场景安全优先，但必须在提交前给用户充分预览和修改机会（● 冲突）

**冲突时默认立场：** 安全优先于效率。但安全确认的交互必须高效——确认弹窗一键可操作，不需要额外滑动或填写。容错路径不得因"界面简洁"而被省略。

**正例：** 转账确认弹窗显示"向张三支付 ¥500"，确认按钮为"确认支付 ¥500"，取消按钮清晰可见。转账失败后提示"收款银行暂时无法处理，请稍后再试或更换银行卡"，提供"重试"和"更换银行卡"两个操作入口。
**反例：** 转账直接执行无确认；或确认弹窗只写"确定？"，不显示金额和收款方。转账失败只显示"操作失败( ERR_TIMEOUT )"，无原因无建议。

## U-008 一致性

**一句话：** 相同功能在不同位置的表现必须一致，核心交互与行业惯例必须对齐。

**判定条件：**
- 同一操作在不同页面使用相同的组件和交互模式
- 同一信息类型在不同页面使用相同的展示格式
- 导航结构在整个产品内保持层级一致
- 术语和文案在同一功能点保持统一
- 颜色语义全局统一（主题色 = 主操作，警告色 = 风险提示）
- 核心交互模式必须与同品类主流产品一致（Jakob's Law：用户大部分时间在使用其他产品，期望你的产品与已有经验一致）
- 不得发明与行业惯例冲突的交互模式（如左滑删除 → 不能改为左滑收藏；底部 Tab 导航 → 不能改为顶部手势导航）
- 当需要偏离行业惯例时，必须在业务原则中显式声明偏离原因和用户学习成本

**适用场景：** 所有页面和组件。

**与谁冲突：**
- U-004 峰终定律：为峰值体验可以打破流程间的一致性
- U-013 信噪比：为降噪移除某类元素可能打破跨页面一致性（○ 偶尔冲突）
- U-014 关键信息前置：不同页面核心信息不同导致布局优先级不同（○ 偶尔冲突）

**冲突时默认立场：** 一致性是默认立场，打破一致性必须有明确的体验收益。每次打破必须记录原因，避免"例外"累积为新的不一致。外部一致性（行业惯例）优先于内部一致性——先与用户已有习惯一致，再追求产品内部统一。

**正例：** 所有业务的"立即购买"按钮使用相同的主题色、圆角、字号，无论在哪个页面。左滑删除、下拉刷新等手势与微信/支付宝一致。
**反例：** 通讯业务用蓝色按钮，信用卡用绿色按钮，跨境用橙色按钮，同样的购买功能三种样式。自创"右滑删除"与行业惯例冲突。

## U-009 识别而非回忆

**一句话：** 所有可选项必须可见或可达，不要求用户记忆之前的选择或猜测未来的路径。

**判定条件：**
- 用户当前选择/状态必须在页面内可见（不需要返回上一步确认）
- 表单已填内容在提交前必须可以回顾
- 下拉选项/筛选条件必须展示当前选中项
- 多步流程中已完成的步骤信息必须可见或可回顾
- 不得要求用户记住上一页的数字、名称或选项来填写当前页
- 多步流程（≥ 3 步）必须展示步骤指示器，显示总步数与当前步骤
- 用户应能从任一步骤预知剩余步骤数量（不需预知内容细节）

**适用场景：** 所有包含选择、填写、多步骤的界面。

**与谁冲突：**
- U-003 席克定律：显示所有信息可能增加选项感知数量（○ 偶尔冲突）
- U-013 信噪比：步骤指示器增加首屏元素，但其信息价值（路径预判）大于视觉成本（○ 偶尔冲突）

**冲突时默认立场：** 识别优先。当信息量与决策简洁性冲突时，通过折叠、渐进式展示来解决——默认展示核心信息，提供"查看详情"入口。

**正例：** 多步汇款流程中，页面顶部显示步骤条"① 填写 → ② 确认 → ③ 支付（当前）"，第 3 步页面显示"收款人：张三 | 金额：¥500"，用户既知道全程有几步，也无需回忆。
**反例：** 多步汇款流程中，无步骤指示器，第 3 步只显示"请确认信息"，用户不知道还有几步、需要返回上一步才能核对收款人。

## U-010 自然语言表达

**一句话：** 使用用户熟悉的语言和概念，不使用技术术语；可交互元素必须通过视觉线索暗示其可操作性。

**判定条件：**
- 界面文案不使用技术术语（如 "session expired" → "登录已过期，请重新登录"）
- 金融数值使用用户视角的表述（如 "手续费 0.1%" → "每转 1000 元收 1 元手续费"）
- 时间使用用户可感知的单位（如 "720h" → "30天"）
- 操作名称使用动词短语（如 "submit" → "提交订单"）
- 错误提示说明原因和解决方案，不只报错码（如 "Error 404" → "页面找不到，请返回首页重试"）
- 可交互元素必须有视觉暗示（Affordance）：按钮必须有明确的可点击样式（填充色/边框/阴影至少一种），纯文本不得表现为按钮
- 交互方式与物理世界或已有数字习惯映射（如开关 → 左右拨动，标签页 → 顶部切换，购物车 → 右上角图标）
- 不可交互元素不得呈现可交互的视觉暗示（如静态文字不得加圆角背景色，避免误导用户点击）

**适用场景：** 所有面向用户的文案和界面元素。

**与谁冲突：**
- U-005 米勒定律：用户语言表述可能比技术术语更长，占用更多信息空间（○ 偶尔冲突）
- U-013 信噪比：Affordance 视觉线索（阴影、边框）增加视觉元素（○ 偶尔冲突，但 Affordance 的信息价值 > 视觉成本）

**冲突时默认立场：** 可理解性优先于简洁性。宁可多用一行文字说清楚，也不用一个技术术语让用户困惑。Affordance 不可省略——缺少操作暗示比多一个视觉元素危害更大。

**正例：** 汇款失败提示"收款银行暂时无法处理，请稍后再试或更换银行卡"，用户知道原因和下一步。CTA 按钮使用填充色+圆角，一眼可识别为可点击元素。
**反例：** 汇款失败提示"系统错误 (ERR_TIMEOUT)"，用户不知道发生了什么。纯文本"点击这里"无任何可点击视觉暗示，用户不会尝试点击。

## U-011 无障碍可达

**一句话：** 所有用户——包括视障、听障、运动障碍和认知障碍——都必须能感知、理解、操作界面。

**判定条件：**

| 子维度 | 判定条件 | 依据标准 |
|---|---|---|
| 色对比度 | 正文文本与背景的对比度 ≥ 4.5:1；大文本（≥ 18px 或 ≥ 14px 加粗）≥ 3:1 | WCAG 2.1 AA 1.4.3 |
| 非色唯一 | 不依赖颜色作为传达信息的唯一手段；错误/成功/警告状态必须有图标或文字辅助 | WCAG 2.1 AA 1.4.1 |
| 文本可缩放 | 页面在 200% 缩放下不得丢失内容或功能 | WCAG 2.1 AA 1.4.4 |
| 键盘可达 | 所有交互元素可通过键盘操作（Tab 导航 + Enter/Space 触发），无键盘陷阱 | WCAG 2.1 AA 2.1.1 |
| ARIA 语义 | 图标按钮、自定义控件必须有 `aria-label` 或 `aria-labelledby`；动态内容更新使用 `aria-live` | WCAG 2.1 AA 4.1.2 |
| 焦点可见 | 所有可聚焦元素必须有可见的焦点指示（`focus-visible` 样式），不得用 `outline: none` 去除焦点框而不提供替代 | WCAG 2.1 AA 2.4.7 |
| 表单标签 | 每个输入字段必须有关联的 `<label>` 或 `aria-label`，不得仅用 placeholder 作为标签 | WCAG 2.1 AA 1.3.1 |
| 标题层级 | 页面标题使用语义化层级（h1 → h2 → h3），不跳级 | WCAG 2.1 AA 1.3.1 |
| 动效敏感 | 必须尊重 `prefers-reduced-motion` 系统偏好；在该偏好下，非功能性动画必须禁用 | WCAG 2.1 AA 2.3.3 |
| 触控目标 | 可触控元素尺寸 ≥ 44px × 44px（与 U-001 费茨定律一致） | WCAG 2.1 AA 2.5.5 |

**适用场景：** 所有面向用户的界面和交互，无例外。

**与谁冲突：**
- U-006 反馈即时性：部分动效反馈（如过渡动画）在 `prefers-reduced-motion` 下需禁用，但不影响即时状态变化的可见性（○ 偶尔冲突）

**冲突时默认立场：** 无障碍是不可妥协的底线，不因任何其他原则而违反。当其他原则与无障碍冲突时，找到同时满足两者的方案，而非牺牲无障碍。

**正例：** 错误状态同时使用红色边框 + 错误图标 + 错误文案；按钮有 `aria-label`；焦点环清晰可见；`prefers-reduced-motion` 下过渡动画关闭但状态变化仍然即时。
**反例：** 错误仅用红色边框标识（色盲用户无法识别）；图标按钮无 `aria-label`（屏幕阅读器读出"按钮"）；`outline: none` 无替代焦点样式。

## U-012 渐进式披露

**一句话：** 默认只展示当前任务必需的信息，细节按需展开。

**判定条件：**
- 首屏可见信息仅包含用户当前步骤的必填/必读内容
- 非必需信息通过"展开""查看详情""更多"等交互按需获取
- 信息层级 ≤ 3 层（首屏默认 → 一级展开 → 二级展开），不得出现第四层
- 折叠内容不得超过该区域展开后总高度的 60%，否则不应折叠
- 关键操作入口（CTA）不得被折叠隐藏

**适用场景：** 信息量超过 7±2 组块的所有页面，尤其是表单、账单详情、产品说明。

**与谁冲突：**
- U-009 识别而非回忆：折叠信息可能导致用户需要回忆之前查看过的内容（● 强冲突）
- U-003 席克定律：展开入口本身是一个额外选项，增加决策节点（○ 偶尔冲突）
- U-006 反馈即时性：展开/折叠的交互增加操作步骤（○ 偶尔冲突）
- U-016 透明与信任：折叠关键过程信息可能削弱用户对结果的信任——关键计算过程不得折叠，次要细节可按需展开（○ 偶尔冲突）

**冲突时默认立场：** 渐进式披露是手段，识别而非回忆是底线——折叠的信息必须可以通过"一步操作"重新获取，不得要求用户记住已折叠的内容。当两者冲突时，优先保证关键信息可见，对次要信息使用渐进披露。

**正例：** 信用卡账单页首屏展示应还金额和还款按钮（核心），点击"交易明细"展开近期交易（二级），交易详情需再点进入（三级）。
**反例：** 账单页把所有 50 条交易记录全部平铺在首屏，用户需要滚动很久才能看到还款按钮。

## U-013 信噪比

**一句话：** 界面上每个元素都必须承载信息价值，无信息价值的元素即为噪声，应当移除。

**判定条件：**
- 每个可见元素必须满足以下至少一项：传达信息 / 支持操作 / 建立层级 / 品牌识别
- 不承载上述任一功能的装饰元素视为噪声，必须移除或弱化
- 背景元素（纹理、渐变、图案）不得降低前景信息的可读性
- 文案字数与信息密度成正比——同义重复、空洞修饰（"非常""超级"）视为噪声
- 数据展示精度与用户决策需求匹配——不需要的精度位数视为噪声（如 "余额 ¥1,234.5678" → "¥1,234.57"）

**适用场景：** 所有页面的视觉设计和文案撰写。

**与谁冲突：**
- U-004 峰终定律：峰值体验可能需要装饰性元素营造情感（○ 偶尔冲突）
- U-008 一致性：为降噪移除某类元素可能打破跨页面一致性（○ 偶尔冲突）
- U-010 自然语言表达：用户习惯的表述可能包含冗余信息（○ 偶尔冲突）
- U-015 用户控制与自由：撤销/退出/返回入口是额外界面元素，但信息价值极高（○ 偶尔冲突）
- U-016 透明与信任：展示计算过程和状态追踪增加界面元素，但透明度的信息价值 > 视觉成本（○ 偶尔冲突）

**冲突时默认立场：** 降噪优先。当装饰性元素有明确的情感收益时可以保留，但必须证明其信息价值（如品牌识别度提升）大于其对信息获取的干扰。无法证明的，移除。

**正例：** 汇款结果页仅展示"已成功汇出 ¥500 → 张三"，无背景装饰，无多余文案。
**反例：** 汇款结果页加了大面积庆祝动效背景、3 行祝福文案、"分享给好友"按钮，关键结果信息反而被淹没。

## U-014 关键信息前置

**一句话：** 最重要的信息放在最前面，细节按重要性递减排列。

**判定条件：**
- 页面首屏必须包含该页面的核心结论或关键数据（结果、金额、状态）
- 信息排列顺序：结果 → 原因 → 细节 → 补充
- 用户无需滚动即可获取该页面的核心信息
- 列表项按信息重要性排序，不得按数据返回顺序排列（除非时间本身就是关键信息）
- 错误/警告信息必须出现在相关区域的顶部，不得埋在段落中间

**适用场景：** 所有信息展示页面，尤其是结果页、账单页、通知页、错误提示页。

**与谁冲突：**
- U-002 格式塔原则：按重要性排列可能打乱按逻辑关系分组（○ 偶尔冲突）
- U-008 一致性：不同页面因核心信息不同，布局优先级不同，可能导致布局不一致（○ 偶尔冲突）

**冲突时默认立场：** 信息优先级高于视觉分组。当格式塔分组与信息重要性排序冲突时，在组内按重要性排序，组间保持格式塔逻辑。

**正例：** 汇款详情页顶部："汇款成功 ¥500 → 张三（中国银行）"，下方依次是汇率、手续费、到账时间。
**反例：** 汇款详情页顶部展示汇款流程时间线，用户需要滚动到页面底部才能看到汇款是否成功。

## U-015 用户控制与自由

**一句话：** 用户必须能随时退出当前状态、撤销已完成操作，不被困在无法回退的流程中。

**判定条件：**
- 每个流程/弹窗/模态必须有明确的"退出"或"取消"入口，且退出路径 ≤ 1 步操作
- 非破坏性操作（编辑、筛选、排序）必须可撤销，撤销入口在操作完成后 ≤ 3 秒内可见
- 破坏性但可恢复操作（删除、关闭）必须提供确认 + 恢复路径（如回收站、30 天恢复期）
- 流程中的任何步骤必须允许返回上一步，不得强制线性推进（除非业务安全要求，如已提交支付）
- 多步流程必须支持"保存草稿"或"暂存"，用户中断后可从断点继续
- 不得出现"死胡同"页面——任何页面至少有一个可操作的出口（返回/取消/关闭）

**适用场景：** 所有流程、弹窗、多步操作和可编辑内容。

**与谁冲突：**
- U-007 安全确认与容错：某些安全场景需要限制用户自由（如支付已提交不可撤销）——此场景安全优先，但必须在提交前给用户充分预览和修改机会（● 冲突）
- U-003 席克定律：撤销/返回入口增加界面选项（○ 偶尔冲突，但控制感 > 选项简洁性）
- U-013 信噪比：退出/撤销入口是额外界面元素，但信息价值极高（○ 偶尔冲突）

**冲突时默认立场：** 用户控制优先于界面简洁。当安全与自由冲突时，提交前充分自由（预览/修改/取消），提交后安全优先（不可撤销），但必须提供恢复路径或补偿方案。

**正例：** 汇款填写页可随时返回修改任一步骤；提交确认弹窗有"返回修改"按钮；支付完成后 5 分钟内可拨打客服撤销。
**反例：** 汇款流程只能向前推进，无法返回上一步修改金额；确认弹窗只有"确认"没有"取消"；操作完成后无任何撤销或补救途径。

## U-016 透明与信任

**一句话：** 用户必须能随时知晓资金状态和流程进展，不因信息缺失而产生不确定感；关键计算过程必须可见，不因结果不明而动摇信任。

**判定条件：**
- 涉及资金的操作，必须展示关键计算过程（汇率、手续费、到账金额），不得只给最终结果
- 等待状态必须说明等待原因和预期时长，不得只显示"处理中"
- 流程中断/异常时，必须说明当前资金状态（已扣款/未扣款）和下一步操作指引
- 用户必须能追溯历史操作的完整记录（交易详情、状态变更时间线）
- 动态变化的信息（汇率、限额）必须标注数据时点和更新频率
- 多步流程中，已完成的步骤结果必须持续可见或可回顾，不得因进入下一步而消失

**适用场景：** 所有涉及资金流动、多步流程、等待状态、动态数据的界面。

**与谁冲突：**
- U-012 渐进式披露：透明要求展示过程细节，渐进披露倾向于折叠——关键计算过程不得折叠，次要细节可按需展开（○ 偶尔冲突）
- U-013 信噪比：展示计算过程和状态追踪增加界面元素，但透明度的信息价值 > 视觉成本（○ 偶尔冲突）

**冲突时默认立场：** 透明优先于简洁。金融场景中，信息缺失导致的信任损失远大于信息丰富造成的认知负担。当两者冲突时，默认展示关键过程，次要细节通过"查看详情"展开。关键计算过程（汇率、手续费、到账金额）始终可见，不得折叠。

**正例：** 汇款确认页展示"汇出金额 ¥500 → 手续费 ¥5 → 汇率 1:7.80 → 到账金额 $63.46"，每项可溯源；等待页展示"银行处理中，预计 10:30 前到账"；超时后展示"处理时间比预期长，您的资金安全，我们正在跟进（已联系收款银行）"。
**反例：** 汇款确认页只展示"将汇出 ¥500"，不展示汇率和手续费；等待页只显示"处理中"，不知道要等多久、钱扣了没；异常后显示"系统繁忙"，用户不知道资金状态。

## 附录：取舍准则（Tradeoff Rules）

当两条通用原则冲突时，按以下优先级取舍。业务原则可覆写此默认优先级，但必须显式声明。

| 优先级 | 原则 | 理由 |
|---|---|---|
| P0 | U-007 安全确认与容错 | 金融产品底线，不可妥协 |
| P0 | U-011 无障碍可达 | 法律合规底线，不可妥协 |
| P1 | U-015 用户控制与自由 | 用户失控 = 信任崩塌 |
| P1 | U-006 反馈即时性 | 无反馈 = 用户失控 |
| P2 | U-010 自然语言表达 | 用户理解是一切的前提 |
| P2 | U-016 透明与信任 | 信息缺失 = 信任崩塌，金融场景透明是不可谈判的底线 |
| P3 | U-009 识别而非回忆 | 不让用户记忆是基本尊重 |
| P4 | U-013 信噪比 | 噪声干扰决策，金融场景信息准确性是生命线 |
| P5 | U-014 关键信息前置 | 关键信息前置决定用户能否快速行动 |
| P6 | U-001 费茨定律 | 可触达性影响操作效率 |
| P7 | U-004 峰终定律 | 体验记忆决定留存 |
| P8 | U-003 席克定律 | 减少决策负担 |
| P9 | U-005 米勒定律 | 减少认知负荷 |
| P9 | U-012 渐进式披露 | 信息管理手段，服务于认知负荷控制 |
| P10 | U-002 格式塔原则 | 视觉组织是手段不是目的 |
| P11 | U-008 一致性 | 一致性服务于体验，不高于体验 |

**使用规则：**
1. 优先级高的原则胜出
2. 优先级相同的原则，由业务原则中的取舍准则裁决
3. 业务原则必须显式声明覆写："本业务覆写通用取舍，P5 > P4，原因是____"
4. 未声明覆写的，按上表默认优先级执行

## 附录：二期候选原则

以下原则一期不入，按踩坑情况决定是否加入二期：

| 编号 | 原则 | 一句话 | 加入条件 |
|---|---|---|---|
| U-017 | 容错性 | 错误操作可撤销，操作后果可预见 | AI 在错误状态设计上出错时 |
| U-018 | 视觉层级 | 每个页面有一个且仅有一个视觉焦点，信息层级 ≤ 3 层 | AI 生成页面缺少焦点时 |
| U-019 | 防呆设计 | 在错误发生之前预防，优于在错误发生后纠正 | AI 在金融操作流程中遗漏防错设计时 |
| U-020 | 冯·雷斯托夫效应 | 在同类元素中，不同的那个会被优先注意 | AI 无法正确引导注意力时 |
| U-021 | 动效规范 | 动效时长 150-300ms，减弱动效偏好下禁用非功能性动画 | AI 动效使用不规范时 |

**加入流程：** 记录 AI 实际出错案例 → 确认现有原则无法覆盖 → 按卡片结构写完整 → 团队评审 → 入库。

### 关键发现

**1. U-016 透明与信任是行业最强共识**

Stripe 和 Wise 都将"透明建立信任"作为金融产品的首要设计原则。Wise 以"真实汇率 + 费用拆解"颠覆传统跨境汇款市场，Stripe 以"展示每一步计算过程"成为金融 SaaS 设计标杆。两条引用：

> "在金融软件中，信任来自透明。展示你的计算过程。" —— Stripe 设计原则

> "透明定价、可见性、控制力可将客户留存率提升至多 60%。" —— Wise《Transparency: a secret growth lever》

**2. U-007/U-010 是 Stripe 错误处理的底层逻辑**

Stripe 的错误处理被业界公认为最佳实践——"explain what happened, why, and what to do next"，正是 U-007（容错恢复）+ U-010（自然语言表达）的组合。

**3. 学术原则在行业体系中隐性嵌入**

U-001（费茨定律）、U-003（席克定律）、U-005（米勒定律）在行业体系中很少以原名出现，但它们的结论被直接写进规范（"44px 触控目标""3 步完成""信息点 ≤ 8 个"）。这说明学术原则是行业规范的底层推导链，即使从业者不自知也在遵循。

**4. 两条原则的行业对标较弱**

U-004 峰终定律和 U-014 关键信息前置在行业体系中鲜有显式对应。这不是原则本身有问题，而是这两条更偏向"设计策略"层面，行业体系通常在"设计模式"而非"价值观"中体现它们。

**5. TDesign 的"包容"比 U-011 更宽**

TDesign 的"包容"不仅指无障碍（U-011），还包含业务场景多样性的包容——"在繁杂的业务场景中寻找共性和特性，确保彼此能灵活地在同一个环境并存"。这对应的是本文 Harness 架构中"通用原则 vs 业务原则"的继承覆写机制，而非单纯的无障碍。
